# Miro Native Mind Map Builder

Собирает **нативный** mind map в Miro (виджет mindmap_node — тот самый, который создаётся кнопкой «Mind map» в приложении) из структурированного дерева: домены → узлы → под-узлы. Работает связкой Miro MCP (чтение/верификация) + computer use (создание узлов клавиатурой в десктопном приложении Miro).

Проверено на реальной задаче: дерево из 105 узлов (6 доменов, 3 уровня вложенности) поверх существующего черновика пользователя, на живой доске с другими людьми онлайн.

## Почему так сложно — ограничения, которые НЕ обойти

Сначала пойми, почему нельзя проще. Мы проверили все пути:

1. **Miro MCP не пишет mindmap.** `layout_create`/`layout_update` поддерживают только FRAME / STICKY / SHAPE / TEXT / CARD / CONNECTOR. Тип `mindmap_node` не создаётся и не читается через `layout_read` (skipped items).
2. **`diagram_create` не умеет mindmap.** Только flowchart / uml_class / uml_sequence / entity_relationship.
3. **REST API v2-experimental (`/mindmap_nodes`) недоступен через MCP-токен.** OAuth-токен MCP-сервера audience-ограничен (`aud: https://mcp.miro.com/`) — api.miro.com его отвергает. Не трать время на выковыривание токена.
4. **Вставка текста из буфера не разбивается на узлы.** Cmd+V многострочного текста с табами в узел вставляет ВСЁ в один узел. Разбиение «строка = узел» в десктопном Miro не работает.
5. **Дерево из SHAPE+CONNECTOR — не то же самое.** Оно строится через API легко, но это не mind map: нет сворачивания веток, нет перетаскивания с re-parent, пользователь просил именно нативный виджет.

Вывод: единственный рабочий путь — **клавиатурная симуляция в приложении Miro через computer use**.

## Что нужно перед стартом

- **Miro MCP** подключён и авторизован (`claude mcp add --transport http miro https://mcp.miro.com -s user`, затем `claude mcp login miro` в интерактивном терминале — OAuth не проходит из неинтерактивного шелла).
- **Computer use MCP** с доступом к приложению Miro (`request_access` с clipboardWrite не обязателен — буфер всё равно не помогает).
- **Десктопное приложение Miro** с открытой доской.
- **Мощная модель + высокий reasoning effort.** Сборка — это ~30 батчей UI-действий с постоянной коррекцией координат по скриншотам. На слабой модели/низком эффорте будет каша: мы собирали на самой сильной доступной модели с максимальным effort, и всё равно ловили и чинили ошибки (список ниже).
- **Структура дерева готова заранее** (выверенный текст каждого узла). Не сочиняй контент на лету — подготовь полный outline до первого клика.

## Клавиатурная механика Miro mind map (проверено)

Работает:
- Клик по узлу → узел выделен.
- **Tab** на выделенном узле → создаёт РЕБЁНКА и входит в режим редактирования текста.
- **Печать текста** (обычный `type`; юникод/кириллица ок) → текст узла.
- **Escape** → выход из редактирования, созданный узел остаётся выделенным.
- **Enter** на выделенном узле → создаёт СОСЕДА (sibling) ниже, в режиме редактирования.

НЕ работает (не пытайся):
- **Стрелки (←↑→↓) НЕ навигируют к родителю/соседям.** Выделение остаётся где было — следующий Enter создаст sibling не там, где ты думаешь.
- **Shift+Tab не поднимает на уровень вверх.**
- **Cmd+V в узел** — вставляет плейн-текст без разбиения (и через synthetic keystroke вообще может не сработать; через меню Edit→Paste сработает, но всё равно в один узел).

Следствие: **DFS-обход без «подъёма» невозможен.** Подъём к родителю делается только МЫШЬЮ — кликом по узлу-родителю.

## Алгоритм сборки

Дерево: домен → core-узлы → под-узлы (smalls).

1. **Подготовь чистое место.** Если во фрейме есть старый черновой контент, который просили заменить — сначала снеси его (см. «Чистка» ниже). Строить в захламлённом фрейме нельзя: узлы наложатся на старьё, клики будут промахиваться.
2. **Один батч = одна ветка без подъёмов.** В `computer_batch`: клик по родителю → Tab → текст → Esc → [Tab → текст → Esc → Enter → текст → Esc → …]. Внутри батча только Tab/Enter/Esc-цепочки — они не требуют координат.
3. **Между батчами — свежий скриншот + zoom.** После каждой вставки Miro пере-лейаутит дерево, и ВСЕ узлы уезжают. Координаты родителя бери только со скриншота, снятого после последнего изменения. Мелкие узлы (<10px) — сначала приблизь зум (кнопкой +/− на панели Miro; Cmd+scroll и ctrl+scroll канвас НЕ зумят).
4. **Тайминги обязательны:** `wait 0.4-0.5` после Tab/Enter (узел создаётся асинхронно), `wait 0.3-0.5` после набора текста перед Escape (иначе узел может не закоммититься), двойной Escape в конце батча.
5. **Микробатч-проверка при сомнении.** Если не уверен в координате родителя — сначала батч из «клик + Tab + текст первого узла + Esc», скриншот, убедись что узел появился В НУЖНОМ месте, потом продолжай Enter-серией.
6. **Верифицируй через API после каждого домена** (см. «Верификация»). Не верь скриншоту — он не показывает parent-связи.

## Известные ошибки (мы их совершили — не повтори)

- **Промах клика по узлу → весь текст улетает в поиск.** Если клик не выделил узел, Tab ничего не создаёт, а набор текста открывает поиск Miro (страницу или боковую панель) — и туда конкатенируются ВСЕ последующие type из батча. Симптом: на экране «Search» / «No results found». Лечение: закрыть поиск, вернуться на доску (кнопка ←), проверить через API, что успело создаться до промаха, продолжить с недостающего. Профилактика — пункт 3 алгоритма.
- **Узел завис в режиме редактирования и не виден через API.** Если последний узел батча остался с курсором (Escape съелся), он не закоммичен — API его не вернёт. Клик по пустому месту канваса коммитит. Всегда проверяй последний узел ветки.
- **Escape иногда меняет зум/вид** (сбрасывает к предыдущему viewport). После Escape не доверяй старым координатам — скриншот.
- **Системные оверлеи перехватывают клики.** Уведомление или Центр уведомлений поверх Miro блокирует клик (computer use откажется кликать в чужое приложение). Лечение: `open_application Miro` снова, или Escape через `osascript System Events`.
- **Пустой узел удаляется сам**, если выйти из редактирования без текста — этим можно пользоваться для отмены случайного Tab.
- **Delete на выделенном узле удаляет ВСЮ ветку** (узел + потомков). Для отката ошибочной ветки это удобно; для точечных правок — осторожно.
- **Другие люди на доске.** Если доска общая — работай строго в своём фрейме. Чужие курсоры двигаются в реальном времени; не выделяй и не двигай чужое. Клик мимо может выделить 200+ объектов (панель «N objects» сверху) — это сигнал немедленно нажать Escape и НЕ жать Delete/Cmd+C.

## Чистка фрейма от старого контента

- Массовое удаление НЕ-mindmap объектов (shapes, connectors, text) быстрее через UI: выделение рамкой (drag по пустому месту) → проверь счётчик «N objects» в панели выделения (он должен совпасть с ожидаемым числом!) → Delete. Marquee выделяет только полностью накрытые объекты.
- Если mindmap пересекается с зоной удаления — сначала временно удали/отодвинь свисающие ветки mindmap, иначе рамка их захватит.
- API-путь (layout_update: old_string=строки DSL → new_string="") работает для точечных удалений, но требует передать точные DSL-строки всех жертв — на сотнях объектов дороже, чем один marquee.

## Ресайз фрейма под выросшее дерево

Mind map растёт от корня во все стороны и вылезает за фрейм. **API-ресайз фрейма с mindmap-детьми падает с «API error»** — только UI: клик по заголовку фрейма → тянуть угловые хэндлы. Тяни строго по одной оси, проверяй перекрытие с соседними фреймами (через `layout_read` фрейма — вернёт актуальные x/y/w/h).

## Верификация (обязательная, после каждого домена и в конце)

`board_list_items` на уровне ДОСКИ (не фрейма!) возвращает mindmap_node с текстом и родителем:
- текст: `data.nodeView.data.content` (HTML, снять теги),
- иерархия: `parent.id` (у корня parent = фрейм, у остальных = родительский узел).

Финальная сверка — скриптом: собрать дерево по parent-chain, распечатать с отступами, сравнить с эталонным outline по числу узлов на каждом уровне и текстам. Не считай задачу сделанной без этой сверки: у нас один узел молча не создался и один лёг не под того родителя — нашлось только скриптом.

## Скорость и бюджет

105 узлов ≈ 25–30 батчей ≈ 40–60 минут работы агента со всеми проверками и восстановлениями после ошибок. Это нормально. Быстрых путей нет (см. «Ограничения»). Если нужен черновик за 2 минуты и нативность не важна — строй SHAPE+CONNECTOR через `layout_create` (группируй элементы с одинаковым набором стилевых параметров в один вызов, font size ≥ 10).
